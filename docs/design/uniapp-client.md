# uniapp 客户端技术方案 —— Android App / 微信小程序 / H5

> 状态：待评审
> 前置讨论结论：uniapp 成为 Home Guardian 的**主客户端**（全功能含管理）；现有 `mobile/`（React）冻结为 Web 体验版，大版本更新时再处理。
> 时序：**用户体系升级（docs/design/home-user-system.md）先行**，本方案 PR1 基于新用户体系开发。

## 1. 定位与目标端

| 项 | 决定 |
|---|---|
| 定位 | 主客户端：日常使用 + 管理功能（设备、告警规则、通知渠道、自动化启停） |
| 目标端优先级 | **Android App 优先** → 微信小程序 → H5 |
| `mobile/` React 版 | 冻结为 Web 体验版，不删除、不再投入新功能 |
| H5 部署 | 构建产物部署到 `/app/`，nginx 加路由；`/mobile/` 不动；稳定后某个大版本再把 `/` 默认跳转切过来 |

### 已知风险：小程序域名限制

微信小程序生产环境仅允许请求 **HTTPS + 已备案域名**，不能用 IP / http。

- 自托管用户（局域网 IP 直连）小程序端基本不可用 → **Android App 是自托管场景主力形态**（无域名限制）
- 小程序定位为"有备案域名用户"的加分项，同一套代码条件编译

## 2. 技术栈

| 层 | 选型 |
|---|---|
| 框架 | uniapp（Vue 3 + TypeScript + Vite，CLI 工程，非 HBuilderX 工程） |
| 状态 | Pinia |
| UI 库 | wot-design-uni |
| 图表 | uCharts（遥测折线） |
| 列表 | z-paging（下拉刷新 / 无限滚动） |
| i18n | 首版即引入框架（vue-i18n），先只填中文，英文文案 PR4 补全 |
| 目录 | `uniapp/` 与 `mobile/` 平级 |
| CI | lint + 各端 build 校验（与 mobile 同款流程） |

Android 清单放开 LAN 明文 HTTP（cleartext）——自托管局域网 `http://192.168.x.x:8787` 刚需。

## 3. 多服务器与扫码

自托管场景 App 需要知道连接哪台服务器：

- **多服务器配置**：本地持久化 N 组 `{名称, baseURL, access/refresh token}`，切换即换上下文（WebSocket 同步重连）
- **添加方式**：扫码为主、手填为辅
- **二维码格式**：`hg://server?url=<baseURL>&name=<名称>`，由 LayUI 后台「生成配置二维码」入口输出（后台小改动，随 PR1）
- 每台服务器独立登录、独立 token 对，互不干扰

## 6. uniPush 推送（PR3）

链路：App 获取推送 cid → 上报后端存储 → 告警触发时后端调 DCloud REST API 推送。

后端配套：

- 新增 `user_push_devices` 表（user_id、cid、平台、最近活跃）
- 新增 `unipush` 通知渠道类型，接入现有 NotificationChannel / NotificationProcess 体系
- DCloud appid / apikey / secret 走 `.env` 配置

**开源摩擦点（写入文档）**：推送凭证绑定 App 的 DCloud appid，自托管用户须自行注册 DCloud 账号并配置凭证。多服务器场景天然合理——每台服务器各自持凭证、各自推送。

App 侧：推送开关、按告警级别过滤（如仅 critical）。
