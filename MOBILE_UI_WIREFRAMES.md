# Home Guardian 移动端页面线框图说明稿

本文档基于 [MOBILE_UI_FUNCTIONS.md](MOBILE_UI_FUNCTIONS.md) 继续拆解，目标不是描述功能清单，而是直接指导另一个 AI 或设计师出页面线框图、低保真稿和首轮视觉稿。

## 1. 使用方式

- 每页按“从上到下”的线框顺序描述
- 每页都包含布局结构、模块关系、视觉重点、交互重点、状态稿要求
- 默认画手机竖屏
- 建议统一使用 8pt 栅格思维，卡片圆角统一，大卡片与普通卡片建立明显层级差

## 2. 全局线框规则

## 2.1 页面基础骨架

除登录页外，绝大多数页面采用以下骨架：

1. 顶部安全区
2. 导航栏或页面标题区
3. 页面主内容区
4. 底部安全区
5. 一级导航页额外带底部 TabBar

## 2.2 页面容器节奏

- 页面左右边距建议统一
- 大模块之间用明显留白分层，而不是仅靠细边框
- 每个页面尽量只有一个主视觉焦点，避免所有卡片同权重
- 列表页强调扫描效率
- 详情页强调信息分层
- 表单页强调填写路径

## 2.3 卡片层级建议

建议将卡片分成 4 层：

1. Hero Card：页面主视觉，用于首页环境、设备空间、告警高亮、主题预览
2. Summary Card：统计摘要和总览信息
3. Action Card：承载按钮、快捷入口、可执行动作
4. List Card：承载明细项、日志、规则列表、渠道列表

## 2.4 通用视觉原则

- 首页偏情绪化和空间感
- 设备与图表页偏数据化
- 告警与命令页偏状态化
- 表单页偏流程化
- 我的页偏轻设置中心

## 2.5 通用状态稿要求

核心页面至少要给出以下其中两类状态稿：

- Loading 骨架态
- Empty 空状态
- Error 异常态
- Success 反馈态
- Confirm 确认弹层态

## 3. 页面线框拆解

## 3.1 登录页

### 页面定位

- 品牌入口页
- 信息简单，但视觉要足够完整，避免像默认后台登录框

### 线框结构

从上到下：

1. 顶部留白区
2. 品牌区
3. 登录表单卡
4. 主按钮区
5. 底部留白区

### 模块说明

#### 品牌区

- 中心对齐
- 包含 Logo 字母标识 HG
- 产品名 Home Guardian
- 一句产品说明
- 可带 2 个轻量标签，如 IoT Dashboard、PWA Ready

#### 登录表单卡

- 一张居中的玻璃态或浅浮层卡片
- 内含两个垂直输入项：用户名、密码
- 输入框高度明显大于后台系统风格

#### 主按钮区

- 一个全宽主按钮
- 与输入框形成一条清晰提交路径

### 视觉重点

- 焦点是品牌和表单卡，不是背景装饰
- 整体要有“家庭科技产品”的柔和感，不要做成工业后台

### 必出状态稿

- 正常态
- 提交中态
- 登录失败反馈态

## 3.2 首页

### 页面定位

- 产品的核心展示页
- 需要一屏建立“家庭空间 + 实时状态 + 快速操作”的印象

### 线框结构

从上到下：

1. 顶部导航区
2. 环境 Hero 主卡
3. 常用场景区
4. 统计卡组
5. 快捷入口宫格
6. 房间区
7. 最近告警区
8. 底部 TabBar

### 模块说明

#### 顶部导航区

- 左侧：页面标题，如“我的家”或“某某的家”
- 下方一行副标题
- 右侧：设置入口按钮

线框表现：

- 不是传统 navbar，像轻型欢迎头
- 文本与按钮并排

#### 环境 Hero 主卡

- 这是首页最大卡片
- 左侧为主温度大数字
- 右侧为 3 个环境指标小块：温度、湿度、空气质量
- 可加入轻天气感文案，如多云、舒适、空气优

线框表现：

- 一个横向大卡
- 左大右小，形成明显主次

#### 常用场景区

- 标题行：常用场景 + 查看自动化
- 下方 4 个场景按钮或小卡
- 建议 2x2 宫格

线框表现：

- 每个场景卡含图标 + 文案
- 按钮外观可略圆润，像可点击的生活模式入口

#### 统计卡组

- 3 张横向统计卡
- 内容：设备总数、在线数、待处理告警

线框表现：

- 三等分或 1 行 3 卡
- 告警卡应有更强视觉强调

#### 快捷入口宫格

- 标题行：快捷入口 + 数据图表
- 下方 6 宫格
- 每个入口：图标 + 文字

线框表现：

- 两行三列
- 强调轻操作和一眼可达，不要做成纯文字列表

#### 房间区

- 标题行：房间 + 全部房间
- 第一张是主房间大卡
- 下方是多个房间 chip 卡

主房间卡内部：

- 房间名
- 设备数
- 在线数
- 代表性指标

#### 最近告警区

- 标题行：告警信息 + 查看全部
- 下方是纵向告警列表
- 每条为横向信息卡：图标、规则名、设备名、房间、时间、箭头

### 视觉重点

- 首页最大焦点必须是环境 Hero 卡
- 第二焦点是快捷入口
- 房间和告警区应更轻，避免抢首页主视觉

### 必出状态稿

- 正常态
- 新手空态：暂无设备 / 引导去设备页
- 告警空态：最近暂无告警

## 3.3 设备列表页

### 页面定位

- 面向查找与筛选
- 视觉目标是“快扫”和“按房间定位”

### 线框结构

从上到下：

1. 页面标题区
2. 搜索框
3. 房间横向入口带
4. 状态 Tab
5. 设备列表区
6. 底部 TabBar

### 模块说明

#### 页面标题区

- 左侧标题“设备”
- 下方副标题说明按房间和状态查看
- 右侧图表入口按钮

#### 搜索框

- 独立一行，宽度占满
- 支持搜索名称、UID、房间

#### 房间横向入口带

- 横向滚动卡片或短卡
- 每个房间卡展示房间名和设备数量
- 当前房间筛选态要有明显高亮

#### 状态 Tab

- 全部
- 在线
- 离线

建议做成悬停感不强、切换清晰的胶囊 Tab

#### 设备列表区

- 优先按网关堆栈展示
- 网关卡作为父卡
- 子设备缩进或嵌套在其下
- 没有网关的设备走普通卡列表

### 单个设备卡线框

从左到右：

1. 设备图标
2. 中间主信息区
3. 右侧状态和箭头

主信息区从上到下：

- 设备名
- 位置 + 类型
- 关键指标摘要

### 视觉重点

- 搜索和筛选比装饰更重要
- 网关与子设备层级必须清楚
- 在线状态需一眼区分

### 必出状态稿

- 正常态
- 房间筛选态
- 空状态：暂无设备

## 3.4 设备详情页

### 页面定位

- 单设备的查看、编辑、控制中心

### 线框结构

从上到下：

1. 顶部导航栏
2. 房间语义 Hero 卡
3. 设备信息卡
4. 实时数据网格
5. 设备属性区
6. 控制区

### 模块说明

#### 顶部导航栏

- 左侧返回
- 中间设备名
- 右侧图表按钮

#### 房间语义 Hero 卡

- 左侧主信息：房间名、设备类型、在线状态、固件版本
- 右侧 2 个 KPI 小块：设备 UID、最后状态/最后上报时间

线框表现：

- 双栏 Hero
- 左栏更宽，右栏是纵向堆叠 KPI 卡

#### 设备信息卡

- 图标
- 设备名
- 状态 Tag
- 详细字段 2 列网格

字段区建议为：

- 类型
- 位置
- 固件版本

#### 实时数据网格

- 标题行：实时数据 + 查看图表
- 下方 2 列或 3 列 metric 卡
- 每卡含图标、指标名、当前值、单位

#### 设备属性区

- 标题
- 属性列表
- 每一项是“键 - 值”双栏
- 底部一个“新增属性”按钮

#### 控制区

仅在执行器设备下出现，内部包括：

- 电源开关行
- 亮度滑杆区
- 刷新状态按钮

### 视觉重点

- Hero 卡是详情页视觉锚点
- 实时数据区是第二焦点
- 控制区要像“操作区”，与信息区风格区分开

### 必出状态稿

- 正常态
- 设备离线态
- 设备不存在态
- 指令发送反馈态

## 3.5 单设备图表页

### 页面定位

- 以单设备历史趋势为核心

### 线框结构

从上到下：

1. 顶部导航栏
2. 页面标题区
3. 指标切换区
4. 时间范围选择区
5. 图表卡

### 模块说明

#### 指标切换区

- 横向 selector
- 只出现该设备已有指标

#### 时间范围选择区

- 1h / 24h / 7d / 30d
- 适合胶囊按钮排布

#### 图表卡

- 标题：当前指标名称
- 副标题：平均值曲线 / 数据范围说明
- 主体是折线图

### 视觉重点

- 图表是唯一主角
- 顶部选择器应该轻，不要压住图表主体

### 必出状态稿

- 首次加载骨架态
- 图表加载态
- 无数据态

## 3.6 全屋图表页

### 页面定位

- 从全局维度看家庭中某个设备的趋势
- 比单设备图表页多一个设备切换维度

### 线框结构

从上到下：

1. 顶部导航栏
2. 标题说明区
3. 全局趋势 Hero 卡
4. 选择器组
5. 图表卡

### 模块说明

#### 全局趋势 Hero 卡

- 标题：全屋趋势分析
- 信息标签：设备数量、当前设备、数据源、时区
- 右侧一个小 KPI 卡显示当前设备

#### 选择器组

- 设备选择器
- 指标选择器
- 时间范围选择器

建议表现为 3 个纵向堆叠的选择卡

#### 图表卡

- 比单设备页更强调整体分析感
- 顶部有图表标题和副标题
- 主体为大图表区域

### 视觉重点

- Hero 卡负责解释“这是全局视角”
- 图表区仍然是主体

### 必出状态稿

- 正常态
- 首次加载态
- 无数据态

## 3.7 告警列表页

### 页面定位

- 告警处理中心
- 是操作页，不只是日志页

### 线框结构

从上到下：

1. 页面标题区
2. 告警摘要卡
3. 状态 Tab 区
4. 告警分组列表
5. 底部 TabBar

### 模块说明

#### 页面标题区

- 左侧标题“告警”
- 副标题强调确认和处理
- 右侧按钮：规则

#### 告警摘要卡

- 展示当前分组数、当前状态筛选
- 作为进入列表前的总览

#### 状态 Tab 区

- 全部
- 触发中
- 已确认
- 已解决

#### 告警分组列表

每张卡从左到右：

1. 状态图标区
2. 文本主体区
3. 数量/时间/箭头区

文本主体区从上到下：

- 规则名
- 状态 Tag
- 设备名 + 房间
- 最近触发时间

展开态应在卡片下方展开子列表：

- 每条具体告警时间
- 触发值
- 状态

#### 分组操作区

建议卡片展开后露出操作按钮：

- 批量确认
- 批量解决

### 视觉重点

- 触发中卡片要最醒目
- 列表扫描时首先看到状态，其次看到规则名和设备名

### 必出状态稿

- 正常态
- 展开态
- 无告警空态
- 批量操作确认弹窗态

## 3.8 告警详情页

### 页面定位

- 单条告警的上下文与处理页

### 线框结构

从上到下：

1. 顶部导航栏
2. 标题区
3. 摘要卡
4. 高亮告警卡
5. 详情信息列表
6. 底部动作区

### 模块说明

#### 摘要卡

- 状态
- 设备名
- 触发时间

#### 高亮告警卡

- 左侧大感叹号图标
- 中间主体：规则名、设备名、房间、相对时间
- 右侧状态标签

#### 详情信息列表

- 规则名称
- 设备
- 触发值
- 触发时间
- 确认时间
- 解决时间
- 消息

#### 底部动作区

- 确认告警
- 标记解决
- 查看设备

建议纵向堆叠或大按钮组

### 视觉重点

- 高亮告警卡是视觉焦点
- 底部动作区应足够明确，方便快速处理

### 必出状态稿

- 触发中态
- 已确认态
- 已解决态

## 3.9 告警规则列表页

### 页面定位

- 规则管理页
- 重点是筛选、启停和批量操作

### 线框结构

从上到下：

1. 顶部导航栏
2. 标题区
3. 统计摘要卡
4. 筛选条
5. 管理动作条
6. 规则列表

### 模块说明

#### 顶部导航栏

- 返回
- 页面名
- 右侧新建按钮

#### 统计摘要卡

- 规则总数
- 启用数量
- 当前视图数量

#### 筛选条

- 全部
- 启用中
- 已停用

#### 管理动作条

普通态：

- 左：当前筛选条数
- 右：进入选择模式按钮

选择模式：

- 左：已选数量
- 右：全选、启用、停用、退出

#### 规则列表

每一项建议从上到下：

- 第一行：规则名、启用状态、选择框
- 第二行：设备名 + 房间
- 第三行：条件摘要
- 第四行：操作按钮行

### 视觉重点

- 规则名和条件摘要要易读
- 管理动作条应悬浮感更强，表明是全局操作层

### 必出状态稿

- 普通态
- 选择模式态
- 空状态

## 3.10 告警规则创建/编辑页

### 页面定位

- 条件配置表单
- 需要做得像“配置规则”，不是简单录入表

### 线框结构

从上到下：

1. 顶部导航栏
2. 标题区
3. 表单 Hero 卡
4. 基础设置卡
5. 触发条件卡
6. 通知渠道卡
7. 提交按钮区

### 模块说明

#### 表单 Hero 卡

- 标题：规则配置
- 副标题：定义设备、指标、阈值和通知渠道
- 辅助信息：设备数、渠道数

#### 基础设置卡

- 规则名称输入
- 设备 Picker
- 遥测指标 Picker

#### 触发条件卡

- 条件 Picker
- 阈值 1
- 阈值 2（条件为 between 才出现）
- 持续时间 Stepper
- 启用开关

#### 通知渠道卡

- 多个渠道标签卡
- 选中态明显高亮

#### 提交按钮区

- 全宽主按钮

### 视觉重点

- 条件卡是页面主体
- 通知渠道区不要做成无层级的复选框表单

### 必出状态稿

- 创建态
- 编辑态
- 条件为 between 时的扩展态

## 3.11 自动化列表页

### 页面定位

- 联动规则管理页
- 管理感比首页场景更强

### 线框结构

从上到下：

1. 页面标题区
2. 自动化摘要卡
3. 筛选条
4. 管理动作条
5. 自动化列表
6. 底部 TabBar

### 模块说明

自动化列表项建议从上到下：

- 第一行：规则名、状态 Tag、启停开关、选择框
- 第二行：触发类型
- 第三行：动作摘要
- 第四行：最近触发时间
- 第五行：编辑 / 删除按钮

### 视觉重点

- 自动化要有“流程感”和“系统响应感”
- 触发类型与动作摘要必须一眼能懂

### 必出状态稿

- 普通态
- 选择模式态
- 空状态

## 3.12 自动化创建/编辑页

### 页面定位

- 半配置型流程编辑页

### 线框结构

从上到下：

1. 顶部导航栏
2. 标题区
3. Hero 说明卡
4. 基本信息卡
5. 触发条件卡
6. 执行动作卡
7. 提交按钮区

### 模块说明

#### 基本信息卡

- 名称
- 触发类型
- 启用开关

#### 触发条件卡

当为 telemetry：

- 设备 Picker
- 指标输入或选择
- 条件输入
- 阈值 Stepper

当为 schedule：

- JSON / cron 文本框

#### 执行动作卡

- 动作类型切换
- 如果是 send_command：目标设备 + 指令名
- 如果是 send_notification：展示一个较简洁的确认说明块

### 视觉重点

- 页面中间要体现“触发 -> 执行”的逻辑链
- 最好画出两个分段卡之间的视觉联系

### 必出状态稿

- 遥测触发态
- 定时触发态
- 发送指令态

## 3.13 命令历史页

### 页面定位

- 指令日志页
- 偏运维、偏追踪

### 线框结构

从上到下：
